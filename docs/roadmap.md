# CapsLock 产品路线图

## 产品定位

CapsLock 面向单个本机用户，是一个按需运行、可恢复、可审计的 CLI/TUI 个人 Agent。它首先可靠地完成工作区任务，再逐步增加公开信息研究、本地工具扩展、长期记忆和可复用工作流。

路线图遵循以下原则：

- 高风险能力必须可见、可拒绝、可审计；文件写入、命令、联网、MCP 和外发数据统一经过权限策略。
- 先完成数据模型、安全边界和评测，再增加自动化、插件和远程能力。
- 本地事实保留路径与行号证据，外部事实保留 URL、抓取时间和来源引用。
- 持久化的用户数据必须可查看、修正、导出和删除，并默认隔离不同工作区。
- 每个小版本只承担一个主要产品风险，避免在同一版本同时引入多个尚未验证的能力面。

## 版本规则

CapsLock 使用语义化版本 `MAJOR.MINOR.PATCH`：

- `PATCH`：缺陷、安全、兼容性、文档和发布修复，不扩大权限或数据范围。
- `MINOR`：向后兼容的新能力；若涉及 SQLite 或配置变更，必须提供自动迁移和回滚说明。
- `MAJOR`：允许调整公开配置、持久化格式或扩展接口，但必须提供迁移工具和明确的兼容窗口。

`capslock/_version.py` 是唯一版本源。Python 包元数据、`capslock --version`、启动界面和发布产物均从该文件读取。正式版本使用相同的 Git tag，例如 `v1.3.0`；普通提交标题不再承担版本标识职责。

## 已完成版本

### 1.0.0：可信只读 Agent

- 只读工作区工具、路径边界、文件大小和二进制文件限制。
- 带路径、行号和稳定 ID 的本地证据。
- SQLite 会话、运行轨迹、恢复和 Git 只读查询。
- 模型工具循环、异常恢复和基础事件审计。

### 1.1.0：受控编辑

- 持久化文件变更提案和统一 diff 预览。
- 逐次批准或拒绝、文件哈希冲突检测和应用审计。
- 对 CapsLock 已应用变更提供可确认撤销。

### 1.2.0：受控执行

- 固定命令模板和逐次审批，不开放任意 Shell。
- 命令超时、取消、输出上限和运行记录。
- 任务状态、上下文压缩以及 token 和费用统计。

### 1.3.0：Web 研究、MCP、权限与 CLI UI

- Tavily 搜索、受 SSRF 保护的公开 URL 抓取、来源审计和不可信内容标记。
- 双层 MCP 注册表和本地 stdio server 的按需连接、工具发现与调用审批。
- `full_access`、`approve_for_me`、`ask_for_approval` 三档权限模式和风险说明。
- `/web`、`/sources`、`/mcp` 等管理入口，以及启动卡片、主题、独立输入区和命令补全。

当前边界：不支持远程 MCP、OAuth、浏览器自动化、自动安装、任意 Shell、后台常驻服务或跨工作区授权。

## 1.3.2：内核解耦与存储演进

### 目标

在不增加工具或权限范围的前提下，拆分 CLI、模型运行时、工具和持久化职责，为记忆与 TUI 开发建立稳定边界。

### 交付结果

- 统一动作状态、结果类型、审批策略与 ID 解析，CLI 和模型工具共用应用层协调器。
- 版本化 SQLite schema；旧动作表迁移前自动备份，迁移失败时事务回滚并停止启动。
- 按领域拆分 repositories、工具适配器、模型协议、CLI 命令声明、输入和渲染组件。
- 保持同步 CLI、现有配置名称、权限范围和主要用户工作流兼容。
- 将 CLI 转换为 package，并进一步拆分组合根、聊天循环、命令分发、动作适配、诊断、输入和渲染。
- 完整命令候选、父命令子树和 alias 补全保持稳定；`/quit` 与 `/exit`、`/session` 与 `/status` 等价。

### 验收结果

- 旧数据库的变更、命令和外部动作可无损迁移，重复迁移不产生额外备份或数据。
- 所有动作只能经过统一审批入口进入执行状态，跨会话和非法状态转换继续被拒绝。
- 当前 65 项测试、ruff、依赖检查与审计、仓库卫生、构建、twine、wheel 安装冒烟和版本一致性校验通过。

## 1.4.0：本地记忆基础

### 目标

先建立可解释、可操作的数据层和手动工作流，不直接启用自动记忆。

### 必须交付

- 版本化 SQLite schema 和迁移框架；迁移前自动备份，失败后保持旧数据可恢复。
- 记忆记录模型：内容、类型、全局/工作区/会话作用域、来源、创建时间、更新时间、置信度和过期时间。
- `/memory list/search/show/add/edit/forget/undo`，所有写操作进入审计轨迹。
- 默认使用本地全文检索；检索结果说明命中的内容、作用域和来源。
- 记忆导出、导入和彻底删除；删除后不得继续出现在检索或模型上下文中。
- 工作区级“禁止记忆”和敏感字段过滤规则。

### 非目标

- 不自动从会话提取记忆，不根据模型判断直接写库。
- 不默认使用外部向量服务，不跨工作区召回项目记忆。

### 验收门槛

- 全局、工作区和会话记忆的隔离测试全部通过。
- 新建、修改、遗忘、撤销、导入和迁移均可审计并可恢复。
- “禁止记忆”的内容不会写入数据库、导出文件或事件日志。

## 1.4.1：会话历史体验

### 目标

在不增加模型工具、权限或数据范围的前提下，让工作区会话历史以自然语言标题呈现，并减少恢复会话时对 Session ID 的依赖。

### 交付结果

- 裸 `capslock` 直接进入新会话，保留 `capslock chat` 兼容入口。
- 会话标题默认取首个非空问题，最长 80 字符；`/rename <title>` 和 `capslock sessions rename` 支持手动重命名。
- `capslock sessions` 以标题为主展示最近更新时间和短 Session ID；标题允许重复，内部 ID 继续保持唯一。
- 裸 `capslock resume` 提供按最近更新时间排序的方向键选择器，展示标题、更新时间和 Session ID；显式完整 ID 或唯一前缀仍可直接恢复。
- `/exit`、`/quit`、EOF 和输入阶段取消时删除完全为空的临时会话，保留已有内容、手动标题或 session 级记忆的会话。
- 工作区 schema 从 v2 升至 v3；迁移前自动备份，并从最早 run 的问题回填旧会话标题。

### 非目标

- 不使用模型生成、改写或归类会话标题，标题也不进入模型上下文。
- 不提供跨工作区会话聚合、全文搜索、归档、导出或批量删除。
- 不以标题替代 Session ID，不要求标题全局唯一。

### 验收结果

- 91 项测试覆盖默认启动、标题生成与重命名、schema v3 回填、空会话清理、短 ID 解析和方向键恢复。
- 恢复选择器在 80 列终端和中英文混排下保持列对齐，表头、标题、更新时间和 Session ID 不重叠。
- Ruff、依赖检查与审计、仓库卫生、构建、twine、版本一致性和 wheel 安装冒烟通过。

## 1.5.1：渐进式 Skill

### 目标

使用 Agent Skills 风格的渐进式 Markdown 工作流建模可复用能力，同时保持现有工具权限边界。

### 交付结果

- `SKILL.md` 包：必填名称和描述、Markdown 正文，以及可选 references/assets/scripts 资源。
- 用户级与工作区级 Skill 注册表，以及明确的覆盖和禁用规则。
- 普通会话使用 16 KiB 名称/描述 catalog，模型通过 `load_skill` 和 `read_skill_resource` 按需加载；消息开头支持 `$skill-name [raw arguments]`。
- `/skills list/show/validate/enable/disable` 管理与检查包；输出恢复为普通文本与现有 citations。
- 加载前进行 frontmatter、路径、大小、UTF-8 和符号链接检查；未知行为字段默认拒绝。
- Skill 调用沿用完整普通工具集及文件、命令、Web 和 MCP 的现有审批与审计，不允许包声明绕过策略。
- 将项目配置、MCP、Skill、本机私有数据和状态统一收口到 `.capslock/`，用户级能力收口到 `CAPSLOCK_HOME`；旧 Skill 路径不再发现或迁移。

### 非目标

- 不提供远程市场、自动下载、自动升级、subagent、hooks 或后台运行。
- 不提供任意脚本执行入口，也不允许 Skill 声明工具、权限或执行模型。

### 验收结果

- 标准 Agent Skills 包可加载；带 Claude 行为扩展字段的包会得到明确错误。
- 恶意或错误 Skill 无法越过工作区、命令、网络和 MCP 边界，同一包在 Linux 与 macOS 上得到一致校验结果。
- catalog 截断、双层覆盖、禁用状态、资源快照、显式/自动调用和三种权限模式均有回归测试。
- schema v5 在自动备份后删除 v1.5.0 开发期 `skill_runs` 及其查询 API，新调用仅写普通 run 审计。

v1.5.0 的强类型 Skill 契约从未正式 tag 或发布，已在 v1.5.1 公开发布前由上述契约取代。

## 1.6.0：上下文智能与受控自动记忆

### 目标

在 1.4.0 的数据基础上增加候选提取和相关召回，但将用户控制置于自动化之前。

### 必须交付

- 会话后提取候选事实、偏好、决策和待办；候选先进入审核队列，不直接成为可信记忆。
- 可配置的 `off`、`review`、`automatic` 三档记忆策略；默认使用 `review`。
- 基于作用域、时效、来源、置信度和相关性的召回排序，并展示记忆为何进入当前上下文。
- 重复合并、冲突标记、过期清理和来源失效处理。
- 可选本地嵌入检索；启用外部嵌入提供方前必须展示将要外发的数据范围。
- 记忆质量评测集，覆盖误记、过期、矛盾、提示注入、敏感信息和跨工作区泄露。

### 非目标

- 不把网页内容、MCP 输出或模型推断无条件视为长期事实。
- 不在用户无感知时将工作区内容发送给新的模型或嵌入提供方。

### 验收门槛

- 自动策略关闭时不产生候选或隐式写入。
- 跨工作区泄露、已删除记忆复现和“禁止记忆”绕过的回归用例必须为零。
- 每条自动记忆都能追溯到来源、提取运行和采用策略。

### 交付结果

- 默认 `review` 的 run 后候选提取、审核队列，以及仅采纳低风险 workspace/session 候选的 `automatic` 策略。
- FTS 与可选本地语义检索的混合排序、4 KiB/top-5 上下文限制及 `/memory context` 解释。
- FastEmbed 可选依赖和仅回环本地 HTTP embeddings；真实外部嵌入 provider 延后至 1.8.0。
- 用户记忆 schema v2、来源失效处理、候选保留清理和导出格式 v2。

## 1.7.0：工作流 TUI

### 目标

让长任务、审批、来源和会话在终端中可持续管理，而不是继续增加斜杠命令堆叠。

### 必须交付

- 流式模型输出和工具状态，明确区分思考、等待审批、运行、成功和失败。
- 计划视图和前台任务队列；支持取消、重试和从最近稳定步骤恢复。
- 集中的审批中心，统一展示文件、命令、Web 和 MCP 动作的风险、载荷与结果。
- 在 1.4.1 的标题、重命名和恢复选择基础上，增加会话搜索、归档、导出和删除。
- 来源、上下文构成、记忆命中和 token/费用的可视化。
- 窄终端、无颜色模式、键盘导航、Ctrl-C 和非交互输出的兼容测试。

### 非目标

- 不引入后台 daemon、定时任务、主动提醒或远程控制。
- 前台进程退出后不继续执行任务。

### 验收门槛

- 中断任意运行不会留下失控子进程或不明确的审批状态。
- 80/120/160 列终端及 `NO_COLOR` 模式均无内容重叠或关键状态丢失。
- 会话导出后可独立检查回答、来源、变更、命令和审批轨迹。

## 1.8.0：模型路由与评测

### 目标

在不改变权限语义的前提下支持多模型角色、故障降级和可重复质量评测。

### 必须交付

- 提供方适配层，至少覆盖 OpenAI 兼容接口，并为后续原生提供方保留稳定协议。
- 为主推理、快速分类、嵌入和视觉预留独立模型角色；每个角色可显式配置。
- 实现真实外部嵌入 provider；启用前预览外发字段、数量和字节范围，并记录用户确认、provider 与数据策略。
- 基于任务类型、上下文上限、预算和可用性的确定性路由，并记录选择原因。
- 超时、限流和服务故障的有界重试与显式降级；不得静默切换到数据策略不同的提供方。
- 按运行、会话和模型统计 token、延迟、错误和费用，支持预算上限。
- 端到端评测框架：工作区问答、编辑、执行、Web、MCP、记忆和 Skill 的固定场景与基线结果。

### 非目标

- 不以不可解释的在线学习自动修改路由策略。
- 不因模型切换扩大工具权限或外发数据范围。

### 验收门槛

- 路由决定、降级原因和费用均可从运行轨迹复现。
- 达到预算上限后停止或请求确认，不继续产生不可见费用。
- 每次发布都运行核心评测，并对质量、延迟或成本回退给出明确结果。

## 1.9.0：生命周期与运维治理

### 目标

完成从首次安装、日常诊断到跨版本升级和数据迁移的完整本地生命周期。

### 必须交付

- `capslock init` 初始化向导：工作区、模型、权限模式和可选外部服务。
- 版本化配置 schema、弃用提示、自动迁移和 `capslock config validate`。
- 凭据提供方接口，支持环境变量和操作系统安全存储；配置与诊断输出始终脱敏。
- `capslock backup/export/import`，覆盖会话、审批、来源、MCP 和记忆数据。
- 增强 `doctor`：检测并给出可执行修复建议，但涉及写入时仍需确认。
- 从 1.3.x 到 1.9.0 的逐版本数据库和配置迁移测试。
- 安装包、升级说明、回滚说明、SBOM 和依赖安全检查。

### 非目标

- 不发布第三方插件市场，不自动安装系统软件。
- 不提供云同步或远程账户体系。

### 验收门槛

- 新用户可从空环境完成配置并通过首个问答。
- 旧版本数据可备份、升级、验证和回滚，不丢失审计链。
- 凭据不会出现在 Git、导出包、SQLite 明文字段或普通日志中。

## 2.0.0：稳定个人 Agent

### 目标

2.0.0 是稳定性和兼容性里程碑，不再集中塞入新的高风险能力。

### 必须交付

- 冻结并文档化 CLI、配置、数据库迁移、Skill manifest 和工具结果协议的兼容承诺。
- 完成安全模型、数据生命周期、故障恢复和权限模式的用户文档。
- 发布受支持的平台与 Python 版本矩阵，以及明确的弃用周期。
- 对安装、升级、恢复、工作区任务、Web、MCP、记忆、Skill 和模型降级执行完整端到端评测。
- 建立发行候选期，关闭阻断级缺陷后再创建 `v2.0.0` tag 和 GitHub Release。

### 非目标

- 不在 2.0.0 临近发布时引入插件市场、后台调度、远程控制或多 Agent。

### 验收门槛

- 从最近两个 minor 版本升级到 2.0.0 均通过自动迁移与回滚测试。
- 核心端到端场景无阻断级缺陷，安全回归和数据隔离测试全部通过。
- 安装包版本、源码版本、文档、tag 和 Release Notes 完全一致。

## 2.x：受条件约束的扩展

以下能力只有在 2.0.0 的安全、迁移和评测指标稳定后才能进入对应版本；若前置条件未满足，应延后而不是压缩验收范围。

### 2.1.0：插件 SDK 与受控分发

- 稳定插件 API、权限清单、兼容性测试和隔离执行策略。
- 先支持本地插件包和显式安装，再评估签名索引或市场。
- 安装、升级、权限变化和卸载都必须确认并保留审计。

### 2.2.0：后台任务与主动提醒

- 独立 worker、任务配额、静默时段、暂停开关和崩溃恢复。
- 定时任务只能引用已批准的工作流；权限升级必须回到前台确认。
- 提供资源、费用和通知频率上限。

### 2.3.0：Dashboard 与远程接入

- 先提供只读运行、来源和审计视图。
- 写操作、审批和任务控制必须使用强认证、会话过期和细粒度授权。
- 默认仅监听本机，不自动暴露公网端口。

### 2.4.0：多模态与文档理解

- PDF、图片和表格通过受限解析器进入统一来源模型。
- OCR、视觉模型调用和文件外发沿用权限、脱敏、预算与审计策略。

### 2.5.0：多 Agent 协作

- 子 Agent 使用隔离工作区、显式任务契约、最小权限和消息审计。
- 父任务只能汇总已验证输出，不能把自身权限隐式传递给子 Agent。
- 在单 Agent 评测和恢复机制稳定前不进入实现阶段。

## 全局完成定义

每个版本只有同时满足以下条件才能标记完成：

1. 设计文档明确目标、非目标、数据模型、权限变化和失败模式。
2. 新增功能包含单元、集成、安全和迁移测试；修复包含可复现的回归测试。
3. 涉及工具权限或数据范围的变化完成威胁模型、审批路径和撤销/恢复验证。
4. `README.md`、工具参考、CHANGELOG、升级说明和示例配置同步更新。
5. ruff、测试、构建、依赖检查、安装冒烟和核心评测全部通过。
6. 更新唯一版本源，创建匹配的 Git tag 和 GitHub Release，并记录已知限制。

## 发布顺序

```text
1.3.0
  -> 1.3.1  发布与质量基线
  -> 1.3.2  内核解耦与存储演进
  -> 1.4.0  本地记忆数据层
  -> 1.4.1  会话历史体验
  -> 1.5.1  渐进式 Skill
  -> 1.6.0  上下文智能与自动记忆
  -> 1.7.0  工作流 TUI
  -> 1.8.0  模型路由、外部嵌入与评测
  -> 1.9.0  生命周期治理
  -> 2.0.0  稳定兼容里程碑
```

不得通过跳过数据迁移、安全评测或发布门槛来提前合并后续版本。版本可因风险和用户反馈延后，也可以发布额外 patch，但 minor 版本的职责边界应保持稳定。
