# CapsLock 产品路线图

## 产品目标与原则

CapsLock 面向单个本机用户，首先是一个按需运行的 CLI/TUI 个人 Agent：既能完成可靠的工作区任务，也能研究公开信息、调用受控外部能力并沉淀长期记忆。

- 高风险能力逐次审批：写入、命令、联网、启动 MCP 与外发数据都必须可见、可拒绝、可审计。
- 先构建平台能力，再增加生态和自动化；新工具不能绕过统一的策略与观测层。
- 本地事实使用路径与行号证据，外部事实使用 URL、抓取时间和来源引用。
- 所有持久化的用户数据应可查看、修正、导出和删除。

## v1.3：Web 研究与 MCP

目标是同时引入受控 Web 研究与 MCP 扩展，保持与本地工具相同的安全模型。

- 增加搜索和网页抓取工具；保存 URL、标题、抓取时间和可引用片段，将网页内容作为不可信输入隔离处理。
- 增加 MCP 注册表、启动/停止、健康检查与延迟工具发现；首批仅支持用户显式配置的本地 stdio server。
- MCP 启动、工具调用、环境变量传递与任何可能外发数据的请求都逐次审批；设置超时、输出上限、密钥脱敏和最小权限范围。
- 提供 `/web`、`/mcp`、`/sources` 等管理入口，能查看来源、已注册 server、工具风险和调用审计。

验收：网页提示注入无法改变权限；MCP 崩溃或异常输出可恢复；所有外部结论可追溯到来源。

## v1.4：自动长期记忆与 Skill

目标是在本机 CLI 中形成可控的个人工作流积累。

- 会话后自动提取候选事实、偏好、任务状态和决策，分别保存为长期记忆、项目记忆和临时上下文，并保存来源与置信度。
- 检索后注入相关记忆，避免将全部历史直接加入模型上下文；提供 `/memory list/search/edit/forget/undo`。
- 新增版本化 Skill 声明：输入/输出、工具依赖、权限需求和触发条件均显式定义；Skill 仍走统一权限与审计层。
- 提供记忆质量评测：误记、过期、跨工作区泄露、撤销和“禁止记忆”的回归用例。

验收：自动记忆可被定位、修改、遗忘和撤销；不同工作区的记忆不会默认混用。

## v2.0：成熟的个人 Agent 体验

目标是把已验证的 CLI 能力整合为完整、稳定、可维护的个人 Agent 产品。

- 强化 TUI：流式过程、计划视图、任务队列、权限面板、来源和上下文可视化、会话搜索与导出。
- 多模型路由与故障降级：按任务选择主模型、快速模型、嵌入模型和视觉模型，并呈现成本与降级原因。
- 完整配置/迁移/诊断体系：初始化向导、版本化配置、凭据管理、数据导入导出、兼容迁移和 `doctor` 修复建议。
- 完善插件 SDK、权限清单、兼容性测试和发布治理；插件安装与升级始终需要用户确认。

验收：新用户可通过向导完成配置；升级不丢失会话、变更、MCP 与记忆；核心能力有稳定的端到端评测基线。

## v2.x：按需求启用的扩展

这些能力不属于当前首发范围，应在核心安全与评测指标稳定后逐项验证。

- 常驻服务、定时任务和主动提醒：采用独立 worker、配额、静默策略和可暂停开关。
- Dashboard 与远程接入：先提供只读运行视图，再逐步支持受认证的审批与任务控制。
- 多 Agent：以隔离工作区、显式任务契约、消息审计和受控汇总为前提，避免子 Agent 获得父任务之外的权限。
- 多模态与文档理解：PDF、图片、表格等先经受限解析器处理，再进入统一的来源和权限模型。

## 发布节奏

- 每个小版本先完成设计文档、实现、回归评测和升级说明，再创建 Git tag 与 Release Notes。
- `README.md` 只维护用户可见的定位、安装、快速开始和安全边界；详细设计与迁移记录放在 `docs/`。
- 每次扩展工具权限或数据范围时，都必须新增威胁模型、审批路径和失败/撤销测试。
