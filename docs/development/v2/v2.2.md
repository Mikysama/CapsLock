# CapsLock v2.2：本机多 Agent 协作

## 范围

v2.2 只支持单机、单层父子 Agent。父 Agent 使用 `delegate_agents` 一次提交一至四个显式任务契约，调度器默认最多并行两个。子 Agent 不能继续委派，不提供 daemon、远程 Agent、跨机器通信或退出后续跑。

## 任务与权限契约

`AgentTaskContract` 包含父 run、目标、不可信输入上下文、路径 allowlist、能力 grants、模型 profile、限制和验证要求。能力不会从父运行复制；缺省子工具目录只有只读类工具，且实际文件访问仍必须命中契约路径 allowlist，空 allowlist 不授予文件访问。文件写入、固定命令、Web 与 MCP 必须显式授予，插件工具不自动进入子目录。

子 Agent 使用独立 `RunInteraction`。交互审批只能由用户完成；无审批器的非交互子动作不会自动执行。父 Agent 的 `full_access` 或当前工作区插件授权不能作为子 Agent 的权限来源。

## 隔离与验证

`AgentWorkspaceManager` 创建不跟随符号链接的文件快照，并排除 `.git`、`.capslock`、任意 `.env.*`、常见凭据/私钥文件和私有环境文件。每个子 Agent 在快照内创建独立 workspace SQLite、memory SQLite 和 session；路径同时受 `WorkspacePolicy` 与契约 allowlist 限制。

`AgentOutputVerifier` 只接受结构化对象，校验必需字段、产物路径、数量、字节上限、实际命令检查结果和 SHA-256。父模型只看到 `ValidatedAgentOutput`；原始自由文本、证据和产物始终是不可信数据。成功任务使用快照基线摘要检测父工作区并发修改，安全复制已验证产物后清理快照；失败、取消或验证失败的快照保留到显式 cleanup。

## 持久化与恢复

workspace schema v5 新增 `agent_tasks`、`agent_workspaces`、`agent_capabilities`、`agent_messages` 和 `agent_outputs`。消息按任务递增序号保存脱敏 payload 与摘要，重复序号由唯一约束拒绝。v4 升级前自动备份；失败时保留备份并拒绝启动。

portable archive 保持 v2，只包含任务、能力、消息和验证输出，不包含临时子工作区。导入的 `created`、`running` 或 `waiting_approval` 子任务转为 `interrupted`，不会恢复历史副作用。

## 配置与验证

```toml
[agents]
enabled = true
max_children = 4
max_concurrency = 2
max_depth = 1
max_child_tool_rounds = 16
```

回归覆盖契约验证、默认只读、二次委派拒绝、并发顺序、兄弟失败隔离、快照隐私、路径越界、摘要篡改、schema v5、生命周期导入和 `/agents` 命令。

本地 Python 3.12 验收为 180 项测试通过；Ruff、repository hygiene、确定性 Agent 评测（质量 1.0、无回归）、依赖审计、wheel/sdist、Twine、版本一致性和 wheel 导入冒烟均通过。正式 tag 前仍需等待 Linux/macOS 远端 CI。
