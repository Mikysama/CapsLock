# CapsLock v2.0：开发过程、架构与迁移

## 目标与约束

v2.0 将 CapsLock 收敛为完全异步、强领域边界的本机 Agent。2.0.0 是稳定性和兼容性版本，不新增插件市场、远程控制、多 Agent、后台 daemon 或多模态能力。

开发过程始终遵守以下约束：

- 业务层不接触 SQLite connection，持久化只通过 repository。
- 模型、工具、动作、workflow 与记忆接口均为 async。
- run 状态、work item、未完成 action/step 和唯一终止事件在同一事务内结算。
- 旧状态不被自动覆盖；升级前备份，失败时保留原数据和恢复入口。
- 文件、命令、Web、MCP、Skill、记忆和外部 embedding 不得绕过权限、脱敏与审计策略。

## 开发过程

### 1. 1.10.1 前置解耦

1.10.1 在不改变 CLI、退出码、配置优先级、数据库 schema、archive、运行事件和权限行为的前提下完成结构收敛：

- 新增中立的 `capslock.ports` 和唯一组合根 `capslock.bootstrap`，解除 runtime/tooling 对 application 与具体 SQLite repository 的依赖。
- 使用共享 `RunInteraction` 传递权限模式和审批回调，删除 Agent 引用回填与模块全局状态。
- 模型运行使用显式 `ModelRunContext`/`ModelRunSession`；Agent 编排拆为 `RunOrchestrator`、`RunEventPublisher` 和 `RunFinalizer`。
- ToolLoop 只依赖 `RunJournal`，模型步骤与工具调用分别执行并生成稳定 checkpoint。
- workflow 状态规则下沉为纯领域策略，repository 负责 CAS、事务和唯一 terminal event。
- lifecycle 拆分为 archive I/O、脱敏、声明式导入规格与双库 `ImportCoordinator`。
- memory repository 拆分为 lifecycle、query、source、settings、audit、candidate、embedding 与 recall 窄入口。
- 配置建立 document reader、validator、migrator、settings resolver 管线；数据库迁移使用 `DatabaseSpec` 与 `MigrationStep`。

这一阶段保留旧构造参数与模型路由方法，只发出默认隐藏且注明 `removed in 2.0.0` 的 `DeprecationWarning`。

### 2. 2.0 接口冻结与清理

2.0.0 删除全部 1.10.1 过渡入口，并将内部调用和测试迁到稳定接口：

| 已删除入口 | 2.0.0 替代 |
| --- | --- |
| `RunContext(repositories=...)` | `tasks=...`、`sources=...`、`actions=...` 等窄端口 |
| `ToolLoop(repositories=...)` | `journal=...` |
| `ModelRouter(repositories=...)` | `audit=...` |
| `ModelRouter.bind_run()` | `router.open_session(ModelRunContext(...))` |
| `ModelRouter.use_role()` | `session.for_role(...)` |
| `ModelRouter.summary(run_id)` | `session.summary()` |
| `max_turns` | `max_tool_rounds` |
| `WorkspaceAgent.ask()` | `WorkspaceAgent.ask_stream()` |

同时确认以下 v1 兼容面已不存在：`SessionStore`、`MemoryStore`、上层 `_connection`、`WorkspaceAnswer`、`last_answer`、`LayoutMigrator`、同步 ToolLoop、Classic UI、`chat`、`ask`、`migrate-layout`、旧 `capslock.tools` 出口，以及 memory export v1/v2 和 session export v1 import。

边界测试直接检查公开签名和模块依赖，防止兼容分支或同步客户端回流。

### 3. 配置与诊断收口

`max_tool_rounds` 是唯一工具轮次配置。`runtime.max_turns` 和 `CAPSLOCK_MAX_TURNS` 在 2.0 中直接失败并显示替代名称，不再警告后继续运行，也不在配置迁移时静默改写。

`capslock init` 直接生成 `config_version = 2`；`doctor` 从配置文档读取并显示真实版本。v0/v1 配置仍可在不含已删除键时备份并迁移，凭据引用与注释保留规则不变。

### 4. 评测与发布门禁

确定性 Agent 评测升级到 schema v2，覆盖 workspace QA、恢复、编辑、命令、Web、MCP、记忆、Skill、模型路由、升级、事务恢复、JSONL 和运行治理。评测环境缺少 pytest 或未生成 JUnit 时返回结构化 `test_runner` 失败，不再以二次解析异常掩盖根因。

CI 在 Linux/macOS × Python 3.12 上执行：

- Ruff format/check 与全量 pytest；
- Agent v2 确定性评测和记忆质量评测；
- `pip check`、`pip-audit`、仓库卫生和完整 Git 历史密钥扫描；
- wheel/sdist、Twine、SBOM、校验和与版本一致性；
- 仓库外 editable/wheel 安装、`init`、`doctor` 和 CLI 版本冒烟。

2.0.0 将最低 Python 版本提升为 3.12。本地 Python 3.12 验收结果为 `163 passed`，Agent 评测质量 `1.0`，记忆评测 `10/10`；wheel/sdist、Twine、依赖审计、版本一致性和隔离 wheel 冒烟通过。

### 5. Python 基线与本地环境

包元数据使用 `requires-python = ">=3.12"`，Python 3.11 不再属于安装或兼容范围。发布 CI 只对 Python 3.12 的 Linux/macOS 组合提供保证；更高版本在进入 CI 矩阵前不视为正式支持。

本地开发统一使用仓库根目录的 `.venv`，不再维护 `.venv-311` 或独立 `.venv-release`。发布验证仍从构建产物创建临时隔离环境，不能依赖本地 `.venv` 中已有的 editable 安装或缓存。

## 当前架构

```text
CLI / TUI / JSONL
        |
bootstrap -> WorkspaceAgent -> ToolLoop -> async ToolRegistry
        |                         |
WorkflowService             ActionCoordinator
        |                         |
WorkspaceRepositories <---- ActionHandlers

MemoryService
   |-- lifecycle / recall / candidates / transfer / embeddings
   `-- MemoryRepositories
```

- `capslock.ports` 是 runtime、tooling 与应用/存储实现之间的中立协议层。
- `bootstrap` 是唯一允许同时了解模型 adapter、应用服务、repository、策略和资源所有权的模块。
- `domain/` 按 sessions、workflow、actions、memory 和 governance 划分，领域类型不依赖 CLI、模型 SDK 或 SQLite。
- `AsyncDatabase` 独占 `aiosqlite.Connection`；复杂状态转换在 repository 内使用同一 connection 和事务。
- `ActionCoordinator` 统一处理文件、命令、Web、MCP 的风险、审批、状态与审计，handler 只负责类型特定的校验与执行。
- memory identity 不保存正文；edit、forget、undo、import 和 adopt 追加不可变 revision，purge 删除正文及依赖但保留无正文审计。

## 稳定协议与事务

### 数据协议

- workspace application ID：`0x434C4B32`，schema v4。
- memory application ID：`0x434C4D32`，schema v3。
- portable archive：v2；JSONL event：v2；配置：v2。
- 只有 application ID 为 0 且无表的空库可以初始化；错误 ID、未知版本或非空未知库直接拒绝。
- 兼容迁移先使用 SQLite backup API 创建一致性备份，再在事务中逐步执行；失败保留备份。

### 运行事务

run 可从 running 进入 waiting_approval、completed、failed、cancelled 或 interrupted。终止事务同时更新 run 和 work item、关闭仍 running 的 step、在失败/取消时关闭未完成 action，并插入唯一递增的 terminal event。重复结算 terminal run 不产生第二个事件。

导入的 running run 转为 interrupted，approved/running action 重置为 pending 并重新审批；已完成副作用只作为历史，不能在目标工作区执行 undo。

## 升级与回滚

### 1.10.0/1.10.1 到 2.0.0

两者已经使用 workspace schema v4、memory schema v3、archive v2 和 config v2，因此 2.0.0 只执行兼容性检查，不创建新 schema migration。升级前停止所有进程并执行：

```bash
capslock backup create
capslock backup verify
capslock doctor --strict
```

如果配置仍包含 `max_turns`，先人工改为 `max_tool_rounds`；环境变量改为 `CAPSLOCK_MAX_TOOL_ROUNDS`。

回滚时停止进程，恢复同一批次的 workspace、memory 与配置备份，再运行 `backup verify` 和 `doctor --strict`。portable import 是逻辑合并，不替代精确回滚。

### 更早版本

- v1.9 配置 v1 可备份迁移到 v2，workspace schema v3 可备份迁移到 v4，memory 保持 v3。
- v1.8 的两套 schema v2 数据库先迁移到 v3；回滚必须同时恢复数据库和配置。
- v1.3.x–v1.7.x 及旧 application ID 不提供直接数据库转换。保留原副本，新建 canonical 工作区，再通过受支持的 session/memory/portable export-import 迁移可验证数据。

旧路径冲突会阻止启动，包括 `capslock.toml`、`capslock.mcp.json`、`.capslock/mcp.local.json`、`.capslock/capslock.sqlite3`、`.capslock/events.jsonl` 和 `.capslock/backups`。不要删除原库来“修复”错误。

canonical 路径为：

- `.capslock/config.toml`
- `.capslock/mcp.json`
- `.capslock/local/mcp.json`
- `.capslock/state/capslock.sqlite3`
- `.capslock/state/events.jsonl`
- `.capslock/skills/<name>/`
- `${CAPSLOCK_HOME:-~/.capslock}/state/memory.sqlite3`

## 安全边界

- 文件写入、命令、Web 和 MCP 都经过 ActionCoordinator、当前权限模式和审计；Skill 不能声明额外权限。
- Web/MCP/记忆/Skill 内容始终是不可信数据，不能覆盖系统提示或授权策略。
- 外部 embedding 启用前展示外发字段、记录数和字节数，并持久化 consent 与 provider/data-policy。
- 记忆按 global/workspace/session 隔离；forget、purge、导出、来源失效和跨工作区召回均有回归测试。
- 模型只向相同 data-policy 的显式候选降级，首个可见流式 delta 后不重放。
