# CapsLock v2 Architecture And Migration

## 目标

v2 将 CapsLock 收敛为完全异步、强领域边界的本机 Agent。核心约束是：业务层不接触 SQLite connection；模型、工具、动作、workflow 与记忆接口均为 async；状态与终止事件通过 repository 事务写入；旧状态绝不被自动覆盖。

## 模块边界

```text
CLI / TUI / JSONL
        |
bootstrap -> WorkspaceAgent -> ToolLoop -> async ToolRegistry
        |                         |
WorkflowService             ActionCoordinator
        |                         |
WorkspaceRepositories <---- ActionHandlers

MemoryService
   |-- lifecycle / recall / candidates / transfer / embeddings
   `-- MemoryRepositories
```

`capslock.ports` 是 runtime、tooling 与具体应用/存储实现之间的中立协议层。
`bootstrap` 是唯一允许同时了解模型 adapter、应用服务、repository、策略和资源所有权的组合根。
运行中的权限模式与审批回调通过共享 `RunInteraction` 传递，不使用 Agent 回填或模块全局状态。

`capslock.bootstrap.open_workspace_application()` 是资源组合根，负责打开两套 repository bundle、创建 AsyncOpenAI adapter、组装 handler、Skill、memory 和 Agent，并在异常或 context 退出时关闭客户端和数据库。`WorkspaceApplication.open()` 保留为兼容 facade，并委托给新组合根。

### Domain

`capslock/domain/` 分为 sessions、workflow、actions 和 memory。领域 dataclass 与 enum 不依赖 CLI、数据库或模型 SDK。

### Storage

`AsyncDatabase` 独占 `aiosqlite.Connection`，提供事务、单条写和锁保护读取。repository 方法是业务层唯一持久化入口。复杂状态转换使用 repository 内部的同一 connection 与事务，不通过多个 public repository 调用拼接原子操作。

工作区 repository bundle：session、workflow、action、task、source、settings、snapshot。

记忆 repository bundle：memory lifecycle、candidate、embedding、recall。

### Runtime

`runtime/model.py` 定义 provider-neutral async model protocol 与 `AsyncOpenAIChatModel`。`ToolLoop` 处理流式 delta、工具参数、usage、checkpoint 与 step 状态。`WorkspaceAgent.ask_stream()` 将持久化事件发布给 CLI；答案只存在 completed event data，不存在 `last_answer` 旁路。

模型调用先创建显式 `ModelRunSession`，其中包含 run、role、limits 和预算基线。Agent 编排由 `RunOrchestrator`、`RunEventPublisher` 与 `RunFinalizer` 分担；ToolLoop 只依赖 `RunJournal`，模型步骤和工具调用分别执行。

### Actions

`ActionCoordinator` 维护 handler registry，执行审批、风险评估和状态转换。handler 分为 file、command、Web、MCP。动作请求和结果是 JSON object；handler 新增字段不会引起 schema subtype 迁移。

### Memory

Memory identity 不保存正文。每次 edit、forget、undo、import 或 adopt 都追加 revision。purge 删除 revision 及其依赖，只留下 identity 和 audit。召回、候选、来源与 embedding 分表，避免生命周期服务继续承担所有查询职责。

repository bundle 同时提供 lifecycle、query、sources、settings 与 audit 窄入口；旧 `memories` 聚合仅作为 2.0 前兼容入口。Embedding backend、外发预览和 consent 由独立策略服务管理。

### Lifecycle 与配置

生命周期模块按 archive I/O、脱敏、声明式导入规格和双库 `ImportCoordinator` 拆分，归档格式保持不变。配置内部按 document reader、validator、migrator、settings resolver 执行；数据库升级由 `DatabaseSpec` 和带 hook 的 `MigrationStep` 声明，通用连接层不再判断 workspace 特定版本。

## 2.0 弃用清单

以下 Python 接口在当前前置重构中仍可用，并发出默认隐藏的 `DeprecationWarning`；最早于 2.0.0 删除：

- `RunContext(repositories=...)`，改用 `tasks=` 与 `sources=`；
- `ToolLoop(repositories=...)`，改用 `journal=`；
- `ModelRouter(repositories=...)`，改用 `audit=`；
- `ModelRouter.bind_run()`、`use_role()` 与 `summary(run_id)`，改用 `ModelRunSession`；
- `max_turns` Python/配置兼容入口，改用 `max_tool_rounds`。

## v1.10.1 前置重构兼容范围

v1.10.1 完成上述边界重构，但不改变 CLI 命令与退出码、配置优先级、SQLite application ID/schema version、portable archive、运行事件、审批权限和既有 Python 导入路径。旧构造参数通过默认隐藏的 `DeprecationWarning` 继续工作；本版本不删除接口、不生成数据库迁移，也不修改用户工作区数据。

静态依赖门禁要求：除 bootstrap 和兼容 facade 外，runtime/tooling 不得导入 application 或具体 storage；domain/ports 不得导入 CLI、模型 SDK 或 SQLite。行为测试继续覆盖 workflow 单事务终止、模型 stream/non-stream 等价、Agent 取消与预算停止、归档失败恢复、记忆 scope/事务以及配置与迁移回滚。

### Presentation

CLI 顶层只有 TUI、exec、resume、sessions、doctor。`cli/views/` 只接收 typed view model 或领域 record，不持有 Agent 或 repository。

## Schema

工作区 application ID 为 `0x434C4B32`，记忆 application ID 为 `0x434C4D32`。schema version 独立管理；v1.10.0 的 workspace schema 为 4，memory schema 为 3。

初始化顺序：

1. 打开 connection，启用 foreign keys 与 busy timeout。
2. 读取 application ID、user version 与现有表。
3. 仅对 application ID 为 0 且无表的空库初始化。
4. schema、application ID 与 user version 在一个事务中提交。
5. 兼容性确认后启用 WAL；记忆库再启用 secure delete。

相同 v2 application ID 的数据库会先使用 SQLite backup API 生成一致性备份，再顺序迁移。v3 增加 lifecycle import 批次、实体映射和 action 导入安全标记；workspace v4 增加 run 治理快照、工具调用指纹和停止原因。迁移失败保留一致性备份；错 application ID、未知 schema version 或其他已有表导致 `IncompatibleDatabaseError`。

## v1.9 到 v1.10 与回滚

普通启动会将配置 v1 保留注释迁移到 v2，并将 workspace schema v3 备份迁移到 v4；memory schema 保持 v3。回滚到 v1.9 前必须停止所有进程，并恢复配置和 workspace schema v3 备份。portable importer 接受 v1.9 archive v1，v1.10 exporter 写 archive v2。

## 状态与事务

run 从 running 进入 waiting_approval、completed、failed、cancelled 或 interrupted。work item 使用 compare-and-set 同步转换。终止事务同时：

- 更新 run 的状态、usage、duration 与 error；
- 更新 work item；
- 关闭仍 running 的 step；
- 在失败或取消时关闭未完成 action；
- 插入唯一递增的 terminal run event。

审批结算检查所有 action。仍有 pending/approved/running 时不结算；否则根据 action 聚合结果完成、失败或取消 workflow。重复结算 terminal run 不产生新事件。

## 旧 v1 数据的手工处理

v1.9 不恢复 v1 application ID 的自动迁移；v1.3.x–v1.7.x 仍按下列步骤保留或人工导出。只有 v1.8 fresh-v2 数据进入自动升级承诺：

1. 停止全部 CapsLock 进程。
2. 备份工作区 `.capslock/`、旧根目录配置和用户记忆路径。
3. 将冲突路径移到工作区之外，保留原始副本。
4. 创建新的 canonical `.capslock/config.toml` 与 MCP 配置。
5. 首次启动 v2，让它创建全新数据库。
6. 通过 v1.8 session export v2、memory export v3 或 v1.9 lifecycle export/import 迁移可验证的数据。

## v1.8 到 v1.9 与回滚

普通启动会先备份并将两套 schema v2 数据库迁移到 v3；无版本配置视为 config v0，保留注释迁移到 `config_version = 1`。回滚到 v1.8 前必须停止所有进程，并同时恢复 workspace、memory 和配置的迁移前备份；v1.8 不能打开 schema v3。

`capslock backup create` 生成本机回滚快照，`verify` 校验成员与 SHA-256，`restore` 在替换前再次备份现状。`capslock export/import` 是经过脱敏的逻辑合并，不替代精确回滚。未完成 lifecycle journal 会阻止普通启动，使用 `capslock doctor --fix` 确认恢复。

会触发拒绝的旧工作区路径包括：

- `capslock.toml`
- `capslock.mcp.json`
- `.capslock/mcp.local.json`
- `.capslock/capslock.sqlite3`
- `.capslock/events.jsonl`
- `.capslock/backups`

canonical 路径：

- `.capslock/config.toml`
- `.capslock/mcp.json`
- `.capslock/local/mcp.json`
- `.capslock/state/capslock.sqlite3`
- `.capslock/state/events.jsonl`
- `.capslock/skills/<name>/`
- `${CAPSLOCK_HOME:-~/.capslock}/state/memory.sqlite3`

如果 canonical 数据库路径已经包含 v1 表，仍会按 application ID 不匹配拒绝。不要删除原库来“修复”错误；先复制到明确的备份位置，再决定是否新建状态。

## 不兼容接口

v2 删除：

- `SessionStore`、`MemoryStore` 与上层 `_connection` 访问；
- 顶层 `capslock.domain` 单文件和 `capslock.tools` 兼容出口；
- 同步 `WorkspaceAgent.ask()` 与同步 ToolLoop；
- `WorkspaceAnswer` 与 `last_answer`；
- `LayoutMigrator` 和旧路径自动发现；
- Classic UI、`chat`、`ask`、`migrate-layout`；
- memory export v1/v2 import 与 session export v1 import。

内部或插件调用方必须改为 domain package、repository bundle、async registry 与 `ask_stream()`。
