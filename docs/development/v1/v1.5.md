# CapsLock v1.5：渐进式 Skill 与统一目录

## 目标与边界

v1.5.1 使用 Agent Skills 风格的 `SKILL.md` 包保存可复用工作流，并通过名称/描述 catalog、按需正文加载和普通 run 执行降低 Skill 系统复杂度。v1.5.0 的强类型 Skill 运行时未正式发布，已被本契约取代。

依赖变更包含 `PyYAML>=6,<7`；已有虚拟环境必须重新安装项目，否则导入 Skill 解析器时会报缺少 `yaml` 模块。CLI 同时支持安装后的 `capslock` 和 `python -m capslock`。

本版本不提供远程市场、安装器、自动升级、subagent、hooks、后台任务或任意脚本执行。Skill 不构成授权，当前 CapsLock 权限模式始终是唯一执行决策来源。

## v1.5.1 更新内容

v1.5.1 是 v1.5 系列最终采用的 Skill 契约。未正式发布的 v1.5.0 强类型实现已被完全替换，不提供旧格式兼容读取或自动转换。

### 新增

- 使用 YAML frontmatter 和 Markdown 正文的 `SKILL.md` 包，支持可选 `references/`、`assets/` 与只读 `scripts/` 资源。
- 普通会话使用最大 16 KiB 的名称/描述 catalog，正文通过 `load_skill(name)` 渐进加载。
- 新增 `read_skill_resource(name, path, start_line, end_line)`，按 run 快照分段读取 UTF-8 资源。
- 支持消息开头的 `$skill-name [raw arguments]` 显式调用，以及交互输入中的 `$` 名称补全。
- 新增 `python -m capslock` 模块入口，与安装后的 `capslock` console script 行为一致。
- `doctor` 显示启用 Skill 数量、catalog 字节数、描述截断状态和无效包数量。

### 变更

- 自动和显式 Skill 都在普通 run 中执行，返回普通文本并沿用 evidence、source 和 memory citations。
- Skill 使用完整普通工具集，执行能力完全由当前 `PermissionMode` 决定；Skill frontmatter 不能声明或扩大权限。
- 依赖由直接使用 `jsonschema`、`packaging` 改为 `PyYAML>=6,<7`。`jsonschema` 仍可能作为 MCP 的传递依赖存在，但不再参与 Skill 校验。
- 工作区 SQLite schema 升至 v5；升级前自动备份 v4 数据库，随后删除旧 `skill_runs`，保留 `skill_settings`。
- 当前工作区的旧 `workspace-summary` 已重生为 `.capslock/skills/my-summary/SKILL.md`。

### 删除

- 删除 `skill.toml`、`instructions.md`、输入/输出 JSON Schema 和 fixture 契约。
- 删除 `SkillManifest`、`SkillAnswer`、`WorkspaceAgent.run_skill`、工具子集 guard 和结构化输出校验。
- 删除 `/skills run`、`key=value` 输入解析和独立 Skill run 审计。
- 删除 `capslock.skills/`、旧用户 Skill 目录的发现和布局迁移支持。
- 删除旧 Skill run 领域类型、repository、CLI 历史查询和 SQLite 表。

### 升级

已有虚拟环境必须重新安装项目以获取 PyYAML 并刷新 CapsLock 版本元数据：

```bash
python -m pip install -e '.[dev]'
capslock --version
python -m capslock --version
capslock doctor
```

如果跳过重新安装，启动时会出现 `ModuleNotFoundError: No module named 'yaml'`。旧 Skill 需要人工重新编写为目录名与 frontmatter `name` 一致的 `SKILL.md` 包。

## 包契约与注册表

包入口是带 YAML frontmatter 的 UTF-8 `SKILL.md`。必填字段为 `name` 和 `description`；可选字段为 `license`、`compatibility` 和字符串 `metadata`。正文必须非空，名称必须与目录一致并符合 Agent Skills 的 1-64 字符命名规则。

包可包含 `references/`、`assets/`、`scripts/` 或其他只读资源。加载器拒绝重复或未知 frontmatter、行为扩展字段、符号链接、非普通文件、目录穿越、超限入口/资源/包和过多资源。`scripts/` 仅是可列出和按 UTF-8 读取的资源，不会注册执行工具。

注册表只合并 `.capslock/skills/` 与 `${CAPSLOCK_HOME:-~/.capslock}/skills/`。项目级覆盖用户级，`skill_settings` 保存当前工作区的启用状态。旧项目/用户 Skill 目录和旧 TOML/Schema 包不再发现或迁移。

## 渐进加载

普通系统上下文只包含有效、启用 Skill 的名称与描述。catalog 按名称排序，预算固定为 16 KiB；名称始终保留，描述按顺序填充，超出预算后使用 name-only，`doctor` 显示数量、字节数、描述截断和无效包数量。

模型可调用两个普通只读工具：

- `load_skill(name)` 返回正文、scope、摘要和资源清单，同一 run 重复加载命中快照缓存。
- `read_skill_resource(name, path, start_line, end_line)` 读取已加载包内的 UTF-8 行范围，拒绝未加载包、目录穿越、二进制、超限内容和越界范围。

资源在首次加载时进入当前 run 的内存快照；后续文件变化只影响下一次加载。工具审计只保存名称、路径、范围、scope、摘要和资源数，不保存正文或资源文本。

## 显式调用与普通运行

只有消息开头的 `$skill-name [raw arguments]` 视为显式调用。未知、禁用或无效包在模型调用前失败。Skill 正文和原始参数使用边界标记作为不可信任务上下文加入普通 run；最终输出是普通文本，并沿用 evidence、source 和 memory citations。

Skill 使用完整普通工具集。文件、命令、Web、MCP 和 memory 继续由 `WorkspacePolicy`、`ActionCoordinator` 和当前 `PermissionMode` 控制，`full_access` 不因 Skill 额外收紧。`.capslock/skills/**` 是持久化指令，Agent 对其修改仍在所有模式下逐次确认。

## CLI 与持久化

CLI 提供 `/skills list/show/validate/enable/disable` 和 `$` 名称补全，不再解析 `key=value` 或提供 `/skills run`。`show` 只展示当前包的 frontmatter、来源、摘要和资源。

SQLite schema 升至 v5：迁移先备份 v4 数据库，再删除 `skill_runs`；领域类型、repository 和 CLI 历史查询同步删除。`skill_settings` 继续保存禁用状态，新 Skill 调用写入现有普通 run、tool call、action、event 和 citation 表。

## 统一目录

项目 config、MCP、Skill、local 和 state 统一位于 `.capslock/`，用户级 Skill 和记忆统一位于 `${CAPSLOCK_HOME:-~/.capslock}`。`capslock migrate-layout` 继续迁移旧 config、MCP、state 和 memory，但不再处理任何旧 Skill 路径。

当前工作区的旧 `workspace-summary` 包已重写为 `.capslock/skills/my-summary/SKILL.md`，不保留旧文件格式。
