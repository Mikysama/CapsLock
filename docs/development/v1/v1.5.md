# CapsLock v1.5：可控 Skill

## 目标与边界

v1.5 将可复用工作流建模为本地、声明式、可校验的 Skill。Skill 只能由用户通过 `/skills run` 显式启动，不向模型暴露隐式触发工具，也不提供脚本执行、远程市场、自动下载、自动升级或后台运行。

## 包契约与注册表

每个 Skill 是独立目录，包含严格字段的 `skill.toml`、UTF-8 `instructions.md`、JSON Schema 2020-12 输入/输出契约和可选 JSON fixtures。manifest 声明名称、SemVer 版本、描述、最低 CapsLock 版本、文件路径、所需工具和权限能力。加载器拒绝未知字段、目录名不一致、绝对/越界/符号链接路径、外部 `$ref`、超限文件、不兼容版本、缺失工具和未声明权限。

用户级包位于 `${CAPSLOCK_HOME:-~/.capslock}/skills/`；工作区包位于 `.capslock/skills/`。注册表按新项目、旧项目、新用户、旧用户发现，同一作用域的新旧同名包只有摘要相同才能合并，项目级继续覆盖用户级。`enable/disable` 状态保存在当前工作区，优先于所有包来源。

fixtures 支持 `input`、`output` 以及 `input_valid`、`output_valid` 预期，可同时覆盖成功契约和预期失败场景。`/skills validate` 完全离线，不执行包内代码或访问网络。

## 运行与安全模型

- `WorkspaceAgent.run_skill` 为每次调用创建普通 run 和关联的 Skill audit，输入与最终输出都必须通过 schema。
- Skill 指令作为低于 CapsLock 系统安全指令的不可信工作流内容注入；模型必须返回无 Markdown 包装的 JSON object。
- `ToolRegistry.subset` 只暴露 manifest 声明的现有工具。`command:<template>` 和 `mcp:<server>:<tool>` 在调用时按参数再次校验。
- 文件、命令、Web 和 MCP 继续经过 `WorkspacePolicy`、`ActionCoordinator`、当前 `PermissionMode` 和原有动作审计；manifest 只能缩小能力。
- Agent 可读取项目 config、MCP 和 Skill，但看不到 local、state、`.env` 或旧运行数据；只允许对 `.capslock/skills/**` 创建文件提案，并且三种权限模式都要求逐次确认。
- 应用、执行或丢弃动作时，Skill 只能引用当前 run 创建的 action。每次工具调用和接受最终输出前都会重载包；禁用、删除或修改会终止运行。

## 持久化与 CLI

工作区 schema v4 新增 `skill_runs` 和 `skill_settings`。Skill audit 保存 run/session、名称、版本、来源、manifest 摘要、声明工具/权限、状态、时间、错误及脱敏输入输出；包删除后仍可通过 `/skills show <run-id>` 检查历史，并关联已有 tool call、action 和 event。

CLI 提供 `/skills list/show/validate/run/enable/disable`。`run` 接受 `key=value`，非字符串按 schema 类型解析 JSON，缺失必填字段交互补齐，未知、重复或不合法输入在调用模型前拒绝。

v1.5 同时统一目录契约：项目配置、MCP、Skill 和状态分别位于 `.capslock/config.toml`、`mcp.json`、`skills/` 和 `state/`，本机私有 MCP 位于 `local/mcp.json`；用户级记忆位于 `${CAPSLOCK_HOME:-~/.capslock}/state/memory.sqlite3`。`capslock migrate-layout` 默认 dry-run，可按 workspace/user/all 显式复制、摘要校验、原子落盘和清理旧源。旧路径在 v1.x 只读兼容，v2.0 移除。

## 验证结果

测试覆盖 manifest/schema/fixture、路径与版本拒绝、四层发现、工作区禁用、输入补齐、强制结构化输出、限定命令、运行中包删除、历史审计、schema v3→v4 备份，以及布局冲突、幂等迁移、目录合并、数据摘要和 Agent 隔离。正式发布仍要求 Linux/macOS、Python 3.11/3.12 CI 矩阵以及 Ruff、依赖审计、构建、twine、wheel 冒烟和版本一致性全部通过。
