# CapsLock v1.6：上下文智能与受控自动记忆

## 目标与边界

v1.6.0 在现有用户级记忆库上增加候选提取、审核/受限自动采纳和可解释召回。默认策略为 `review`；文件、命令、Web、MCP 与 Skill 权限模型不变。

每个成功 run 在回答落库后使用当前聊天模型进行一次无工具提取。只有用户直接陈述的 fact、preference、decision 和 todo 可以成为候选；助手、Skill、工具、Web 和 MCP 内容只作为不可信上下文，不能成为自动记忆来源。提取失败不会使主回答失败，消耗计入原 run。

## 策略与审核

- `off`：不调用提取模型，不产生候选或隐式写入；手动记忆仍可用。
- `review`：默认值。合格候选进入审核队列，不修改可信记忆。
- `automatic`：只自动采纳置信度不低于 0.90、用户直接陈述、workspace/session 作用域、无敏感脱敏、无冲突且来源有效的候选。

global、模型推断、外部事实、冲突或检查不明确的候选始终要求人工审核。自动重复项只增加来源，不覆盖正文。所有自动记忆保留 extraction ID、源 run、模型、prompt 版本和采用策略。

## 召回与嵌入

自动召回合并 FTS 与可选语义排名，再按相关性、作用域、置信度、时效和来源有效性排序。每轮最多注入 5 条、4 KiB，并以不可信 JSON 数据区块提供给模型。回答显示召回摘要，`/memory context` 展示完整得分因素。

嵌入默认关闭。`fastembed` 通过 `local-embeddings` extra 安装；首次启用会确认模型下载。本地 HTTP 后端使用 OpenAI-compatible `/embeddings`，只接受解析到 `localhost`、`127.0.0.1` 或 `::1` 的端点，禁止重定向。任何嵌入错误都会退化到 FTS。

真实外部嵌入 provider、外发预览、确认与审计延后到 v1.8.0。

## 数据升级与回滚

用户记忆 schema 从 v1 升至 v2，增加候选、提取、多来源、向量和召回记录。升级前在用户记忆目录的 `backups/` 自动创建 v1 备份，失败则事务回滚。工作区 schema 保持 v5。

导出格式升至 v2并增加 origin/provenance，可选择包含候选；导入继续接受 v1。降级到 v1.5.1 前必须退出 CapsLock，并用迁移备份恢复用户记忆 schema v1。

## 验证

测试覆盖策略矩阵、严格提取、敏感信息、重复/冲突、来源失效、跨工作区隔离、召回解释、两种嵌入后端和 v1→v2 迁移。发布前还需通过 Ruff、完整 pytest、依赖检查与审计、构建、Twine、wheel 安装和版本一致性校验。
