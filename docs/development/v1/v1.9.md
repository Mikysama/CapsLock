# CapsLock v1.9：生命周期与运维治理

## 目标与边界

v1.9.0 在 v1.8 fresh-v2 基线上补齐首次初始化、版本化配置、OS 凭据、本机回滚快照、portable 数据合并和结构化诊断。本版本只保证 v1.8 到 v1.9 的自动升级，不转换 v1.3.x–v1.7.x application ID，也不提供云同步、账户或系统软件安装。

## 配置与凭据

`config_version = 1` 是公开配置版本。v1.8 无版本 TOML 在 `.capslock/state/backups/` 留存副本后通过 tomlkit 保留注释迁移；`api_key_env` 转为 `credential = "env:NAME"`。另支持 `keyring:NAME`，凭据值只通过环境或 OS keyring 解析，不写入 TOML、SQLite、事件或普通输出。

`capslock init` 在临时内容通过 schema 校验后原子写入，支持 TTY 向导和完整非交互参数。已有配置默认只验证，`--update` 先备份再更新；网络测试必须显式指定 `--check-provider`。

## 数据生命周期

workspace 和 memory schema 升至 v3，增加 `lifecycle_imports` 与 `lifecycle_import_items`；action 增加 import provenance、historical-only 和 reapproval 状态。portable archive 使用版本化 manifest、archive ID、成员清单和 SHA-256，拒绝路径穿越、符号链接、特殊文件、超限和伪造清单。

导入对同 ID/同内容记录幂等跳过；同 ID/不同内容以 archive ID 确定性重映射，并重写关系。workspace/session memory 绑定目标 workspace；global memory 只有显式导出才进入包。MCP env、模型配置、keyring、embedding vector/consent 和缓存不进入 portable archive。

导入 running work 变为 interrupted，queued work 需要 `/queue start`，approved/running action 重置为 pending 并重新校验。历史副作用仅用于审计，不能在目标工作区撤销。

## 诊断与恢复

Doctor 在普通 Settings 解析前执行，支持文本/JSON、strict、显式网络检查和确认式修复。它检查配置、credential reference、数据库 ID/schema/integrity、权限、MCP、Skill 和 lifecycle journal。写修复必须使用 `--fix` 并确认；`--yes` 仅用于显式自动化。

本机 backup 使用 SQLite backup API 保存双库和受管文件；凭据与 MCP env 值排除。导入/恢复先创建持久回滚快照并写 journal，未完成 journal 会阻止普通 Agent 启动。
