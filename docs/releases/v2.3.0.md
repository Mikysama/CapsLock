# CapsLock v2.3.0

CapsLock 2.3.0 将工具系统升级为参数感知、可流式、可中断、可持久化暂停且支持动态发现的 Tool Runtime v2。新运行时继续使用 Action 审批、执行前 revalidate、undo、插件沙箱和 capability broker，同时将模型工具统一为直接能力名称。

## Tool Runtime v2

- `ToolContract`、`ToolDefinition` 和 `ResolvedToolPolicy` 分别描述 schema、执行实现与实际参数解析后的权限/取消策略。输入和结果数据在边界使用 JSON Schema Draft 2020-12 校验。
- `ToolPause` 将审批和用户输入从终态结果分离。invocation、run step、run 与 work item 保存 continuation checkpoint，重启后可恢复同一次调用而不重复执行。
- 只读、并发安全且不修改上下文的调用可并发，结果仍按模型 tool-call 顺序提交。普通异常只失败当前调用，只有显式 `fail_fast` 才取消兄弟任务。
- 执行状态与结果 delivery 状态分离。大结果使用稳定预览和 content-addressed artifact，artifact 写入失败或截断不会否定已完成的副作用。

## 直接能力工具

- 文件与检索：`write_file`、`glob_files`、增强的 `read_file`，以及基于 ripgrep 的搜索后端。文件写入继续生成 Action、diff、hash revalidation 和 undo。
- 交互与任务：可持久化的 `ask_user`，以及 `create_task`、`list_tasks`、`get_task`、`update_task` 和依赖环检测。
- 代码语义：定义、引用、实现、符号、调用者与被调用者查询，使用已安装或显式配置的 LSP server。
- 资源与文档：`list_mcp_resources`、`read_mcp_resource`、`read_pdf`、`read_notebook` 和可逆的 `edit_notebook` Action。
- 上下文与协作：session 级 Worktree 创建/退出、异步 `delegate_agents`、子任务查询与停止。

旧 `propose_*`、`task_list_update` 和 `task_status_update` 工具不再注册，也不提供名称兼容层。MCP 和插件工具分别使用 `mcp__<server>__<tool>` 和 `plugin__<plugin>__<tool>`。

## Shell 与动态集成

- `shell` 在 Linux Bubblewrap 或 macOS sandbox-exec 中执行，默认只写工作区且断网。越界、网络或宿主执行必须单独授权，沙箱后端不可用时 fail closed。
- 确定性 Shell 解析器 hard deny 提权、设备访问、沙箱逃逸和宽泛递归删除。快速模型分类器只能在默认断网沙箱中、确定性规则可分类且置信度不低于 0.95 时自动放行。
- 后台 Shell 进程按 session 隔离，支持有界 stdout/stderr、超时、`process_output` 和 `process_stop`。
- MCP 使用唯一的受管长连接路径，支持 tools/resources 发现、list-changed、重连、取消和工作区切换，不再回退到单次 stdio 连接。
- `search_tools` 按需发现 deferred 工具；插件、MCP、LSP 和大 schema 工具可在运行中动态加入 Catalog，常驻 schema 受 8,000-token 预算限制。

## 模块架构

- `runtime/` 只包含模型、Agent、Run 和 ToolLoop 内核；`lsp/`、`mcp/`、`shell/` 各自拥有协议、沙箱、进程和生命周期管理。
- `ports/` 定义中立 LSP/MCP 接口，tooling 和 Action handler 不导入具体 manager。
- `ToolCatalog` 与 `ToolExecutor` 分离，每个 `tooling/tools/*` 模块同时拥有 schema、policy resolver、execute/resume 和 factory，避免集中声明与执行适配的双重维护。
- `composition/` 构造 integrations、tools、actions 和 collaboration；`bootstrap.py` 只协调顶层资源所有权与 active-workspace 切换。
- 删除旧 `*_runtime.py`、异步 Catalog/adapter 与重复 Shell 安全规则，不保留内部导入 shim。

## 升级与兼容性

- 当前格式为 config 5、workspace schema 8、memory schema 3、portable archive 3、JSONL schema 3 和 plugin manifest/protocol/grant 4。
- workspace schema v6/v7 启动时先 checkpoint WAL，再通过 SQLite backup API 写入 `.capslock/state/backups/`，最后在事务中升级。config v3/v4 同样先备份再原子转换为 v5。
- 迁移失败会回滚并保留原数据与备份。v3 插件不自动改写，而是禁用并显示明确的 v4 升级错误。
- Python Tool API、旧工具名称和旧内部导入路径不提供兼容层。

## 发布验证

2.3.0 发布候选已通过 compileall、Ruff、全量 pytest、发布版本一致性、repository hygiene 与构建产物检查。测试覆盖工具契约/生命周期、权限、暂停恢复、迁移、Shell 安全边界、MCP/LSP/插件动态生命周期与架构 import boundary。
