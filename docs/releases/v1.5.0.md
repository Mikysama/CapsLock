# CapsLock v1.5.0 发布说明

v1.5.0 增加本地声明式 Skill，让用户可以保存并显式运行可复用工作流，同时继续沿用现有文件、命令、Web 和 MCP 权限边界。

## 主要更新

- Skill 包使用 `skill.toml`、`instructions.md`、JSON 输入/输出 schema 和可选 fixtures；加载前严格检查字段、版本、路径、工具和权限依赖。
- 用户级 `${CAPSLOCK_HOME:-~/.capslock}/skills/` 与工作区 `.capslock/skills/` 合并，同名工作区包覆盖用户包；`/skills enable|disable` 只影响当前工作区。
- `/skills run <name> [key=value ...]` 是唯一运行入口，缺少的必填值会交互询问，输出必须是符合 schema 的 JSON object。
- `/skills list/show/validate` 展示来源和状态、执行离线兼容检查，并可在包删除后按 run ID 查看历史审计。
- 附带 `examples/skills/workspace-summary/` 只读示例，可复制到工作区注册表后使用。
- 项目 config、MCP、Skill、私有配置和状态统一进入 `.capslock/`，用户级能力统一进入 `CAPSLOCK_HOME`；新增默认 dry-run 的 `capslock migrate-layout`。

## 权限与安全

- Skill 只能看到 manifest 声明的现有模型工具；限定命令模板和 MCP server/tool 会在每次调用时重新验证。
- Skill 指令视为不可信内容，不能覆盖系统指令、当前权限模式、工作区路径边界、固定命令模板、SSRF 防护或 MCP 白名单。
- 文件、命令、Web 和 MCP 动作继续使用现有提案、审批、执行和审计链路；跨 Skill run 的 action ID 被拒绝。
- 每次工具调用和接受输出前验证包摘要；运行期间禁用、删除或修改 Skill 会使本次运行失败。
- 包只能引用自身常规文件和内部 JSON Schema `$ref`，拒绝符号链接、目录穿越和网络引用。
- Agent 不读取 `.env`、`.capslock/local/`、`.capslock/state/` 或旧运行文件，只能对 `.capslock/skills/**` 提出写入，且 `full_access` 也不能跳过逐次确认。

## 升级与回滚

先运行 `capslock migrate-layout` 预览，再运行 `capslock migrate-layout --apply`（非交互环境使用 `--apply --yes`）。迁移在加载 `.env`、配置、模型或 SQLite 前执行，采用预检、临时复制、摘要校验、原子改名和最后清理旧源的流程。

首次打开迁移后的 v1.4.1 工作区数据库时，CapsLock 在 `.capslock/state/backups/` 创建 schema v3 备份，再升级到 schema v4 并增加 `skill_runs` 与 `skill_settings`。迁移失败会回滚并保留备份，重复启动不会重复备份。

v1.4.1 不支持读取 schema v4。回滚前必须退出 CapsLock，并从 `.capslock/state/backups/` 恢复对应 schema v3 数据库。v1.x 保持旧目录只读兼容并提示迁移，兼容代码计划在 v2.0 移除。

## 已知边界

- 不提供远程市场、自动安装、自动升级、隐式触发、任意脚本或后台 Skill。
- Skill 输出只支持 JSON object，不接受自由文本或 Markdown code fence。
- `enable/disable` 是当前工作区本地状态，不修改或删除原始包。
