# CapsLock v1.3.2 发布说明

v1.3.2 是一次保持用户工作流兼容的内部架构升级。该版本不增加工具、权限范围、远程 MCP 或后台能力，重点统一动作生命周期、引入可恢复的数据库迁移，并拆分模型运行时、工具和 CLI 边界。

## 主要更新

### 统一动作与应用层

- 新增 `ActionType`、`ActionStatus`、`ActionResultKind` 和统一动作信息模型。
- 文件、命令、Web 与 MCP 动作共享 `pending → approved → running → completed/failed/cancelled` 生命周期；拒绝动作进入 `rejected`。
- 新增 `ActionCoordinator`，统一提案、风险评估、自动审批、人工批准、拒绝、执行、ID 解析和文件撤销。
- 文件撤销保留 `completed` 状态，并将结果类型从 `applied` 更新为 `undone`。

### 存储与迁移

- SQLite schema 升级为 v1，公共生命周期写入 `actions` 主表，领域载荷分别写入文件、命令和外部动作明细表。
- `SessionStore` 改为数据库与 repository facade，连接、迁移以及 session、run、action、source、task、settings 数据访问具有独立边界。
- 首次打开 schema v0 数据库前，使用 SQLite backup API 在 `.capslock/backups/` 创建带版本和时间戳的备份。
- 迁移在单个事务中完成，并校验记录数和动作 ID 唯一性；重复 ID、校验失败或迁移异常会回滚并停止启动。
- 重复启动不会再次迁移，也不会为已升级数据库重复创建备份。

旧动作状态按以下规则保留语义：

| 旧数据 | v1.3.2 状态与结果 |
| --- | --- |
| 文件 `applied` | `completed / applied` |
| 文件 `undone` | `completed / undone` |
| 文件 `discarded` | `rejected` |
| 命令成功 | `completed / exit_zero` |
| 命令非零退出 | `failed / nonzero_exit` |
| 命令超时 | `failed / timeout` |
| 命令执行异常 | `failed / execution_error` |
| 外部动作成功 | `completed / success` |
| 外部动作失败 | `failed / execution_error` |

### 模型、工具与资源生命周期

- 引入同步 `ChatModel` 协议以及内部请求、响应、工具调用和用量类型；OpenAI 兼容 SDK 通过 adapter 接入。
- 将上下文构建、工具循环、引用解析和运行记录从 `WorkspaceAgent.ask` 拆为独立组件。
- 回答只包含当前 run 产生的事件，不再混入先前运行事件。
- 工具拆分为注册表、工作区/Git、任务/来源和动作适配器，风险与审批策略不再由每个工具重复声明。
- 模型、运行时、命令、Web 和 MCP 配置改为内部逻辑分组；环境变量和 TOML 名称保持不变。
- 文本文件类型、风险判断和敏感信息脱敏集中维护。
- SQLite、模型 client、内部 HTTP client 和 MCP 资源具有正常与异常路径上的明确关闭流程。

### CLI 架构与交互

- `capslock/cli.py` 已转换为 `capslock/cli/` package；console script `capslock`、`capslock.cli:main` 和 `python -m capslock.cli` 入口保持可用。
- `app.py` 只负责参数解析、配置加载、依赖装配、资源生命周期和顶层错误映射。
- 聊天循环、声明式命令目录、斜杠命令分发、动作适配、诊断、prompt-toolkit 输入和 Rich 渲染分别位于独立模块。
- 使用 `CliContext` 显式传递终端与 Agent，移除 CLI 子模块对全局 `console` 的依赖。
- Rich 对象构造集中到渲染层；动作模块只负责查询、确认和调用 `ActionCoordinator`。
- 用户输入区增加与上边框等宽的下边框，原有帮助栏保留在下边框之后。
- 完整叶子命令（例如 `/exit`、`/status`）输入后仍保持候选可见；完整父命令（例如 `/permissions`、`/mcp`）会展开其子命令。
- 命令 alias 进入同一补全和描述目录；`/quit` 与 `/exit` 等价，`/session` 与 `/status` 等价。

## 兼容性

- CLI 参数、斜杠命令、配置变量、TOML 名称、权限语义和主要工作流保持兼容。
- 工作区数据库路径仍为 `.capslock/capslock.sqlite3`，旧数据库自动升级且不会静默丢弃历史记录。
- CLI 列表统一显示通用动作状态，并在适用时附带 `result_kind`。
- 不兼容未公开的 Python 私有函数；相关测试已迁移到应用层和 CLI 模块边界。

## 验证

当前本地 v1.3.2 基线包括：

- 65 项 pytest 测试通过，覆盖迁移、状态转换、动作执行、模型 adapter、事件作用域、资源关闭和 CLI 交互回归。
- Ruff、仓库卫生和 `git diff --check` 通过。
- `pip check` 无依赖冲突；`pip-audit` 无已知漏洞，仅按预期跳过未发布到 PyPI 的本地 `capslock 1.3.2` 包。
- wheel 与 sdist 隔离构建、`twine check`、版本一致性和 wheel 模块布局校验通过。
- wheel 安装后可导入 `capslock.cli`，并可通过模块入口输出 `capslock 1.3.2`。

正式发布前仍应完成远端完整 CI 和凭据检查。正式发布使用 `v1.3.2` tag，将 wheel 与 sdist 附加到对应 GitHub Release，不上传 PyPI。
