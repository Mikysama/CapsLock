# CapsLock v1.3.1 发布说明

v1.3.1 将 v1.3.0 整理为可重复安装和发布的 Linux/macOS 基线，不增加工具、权限或持久化数据范围。

## 安装

需要 Python 3.11 或 3.12：

```bash
python -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install capslock-1.3.1-py3-none-any.whl
capslock --version
capslock --workspace /path/to/workspace doctor
```

正式 wheel 和 sdist 由 `v1.3.1` tag 对应的 GitHub Release 提供，不发布到 PyPI。

## 安全处置

仓库曾跟踪 `.env.swp`，旧提交还包含 `.capslock` 运行数据。任何曾写入这些文件的 DeepSeek、OpenAI、Tavily 或 MCP 凭据都必须视为已经暴露；删除文件不能使凭据重新安全。

发布负责人必须按以下顺序处理：

1. 在对应服务商控制台撤销旧凭据，创建权限最小的新凭据，并只将新值保存到本机 `.env` 或密钥存储中。
2. 暂停仓库写入并备份远端引用，然后用 `git-filter-repo` 从全部引用删除敏感产物：

   ```bash
   git filter-repo --force \
     --path .env.swp \
     --path .capslock/capslock.sqlite3 \
     --path .capslock/events.jsonl \
     --invert-paths
   ```

3. 对重写后的完整历史运行 Gitleaks；只有扫描通过且所有旧凭据已撤销后才能强推分支和标签。
4. 强推后关闭写入冻结，要求所有协作者删除旧 clone 并重新克隆。不得把旧分支、stash 或 tag 推回远端。
5. 在 GitHub Secret Scanning 中将旧告警标记为 revoked；如果敏感对象仍可通过托管平台缓存访问，联系 GitHub Support 清除缓存。

轮换记录只能保存凭据名称、服务商、撤销时间和负责人，不得记录凭据值。

## 发布验证

创建 tag 前，从干净 checkout 执行：

```bash
python -m pip install -e '.[dev]'
python -m ruff check .
python -m pytest -q
python scripts/check_repository.py
rm -rf build dist capslock.egg-info
python -m build
python -m twine check dist/*
python scripts/verify_release.py \
  --tag v1.3.1 \
  --dist dist/capslock-1.3.1-py3-none-any.whl \
  --dist dist/capslock-1.3.1.tar.gz
```

GitHub Actions 会在 Ubuntu 与 macOS 的 Python 3.11/3.12 环境中重复测试、构建、依赖检查以及 editable/wheel 安装冒烟测试。所有矩阵与安全任务通过后，tag job 才会创建 GitHub Release。

## 已知边界

- v1.3.1 不提供远程 MCP、OAuth、任意 Shell、后台任务、长期记忆或 Skill。
- `doctor` 只报告当前配置和本地能力，不连接模型或第三方服务。
- 历史重写会改变既有提交哈希；旧 clone 不能继续用于推送。
