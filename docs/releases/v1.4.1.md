# CapsLock v1.4.1 发布说明

v1.4.1 改进本地会话历史管理：启动命令更直接，会话以自然语言标题呈现，并可从终端列表中选择恢复。该版本不扩大模型工具、文件权限、网络或 MCP 边界。

## 主要更新

### 会话标题与历史

- 新会话在收到首个非空问题后，以该问题生成最长 80 字符的标题；完整问题仍保存在 run 记录中。
- `/rename <title>` 重命名当前会话，`capslock sessions rename <session-id-or-prefix> <title>` 重命名历史会话。
- 手动标题不会被后续问题覆盖；标题允许重复，Session ID 继续作为稳定唯一标识。
- `capslock sessions` 以标题为主，显示最近更新时间和短 Session ID。

### 启动与恢复

- 裸 `capslock` 直接进入新会话，`capslock chat` 保持兼容。
- 裸 `capslock resume` 按最近更新时间列出会话，支持方向键选择和 Enter 恢复；默认显示 20 条，可用 `--limit` 调整。
- `capslock resume <session-id-or-prefix>` 继续支持脚本化或直接恢复。
- 恢复选择器按终端显示宽度对齐中英文标题；标准终端显示完整 ID，极窄终端显示可用的短前缀。

### 空会话清理

- `/exit`、`/quit`、EOF、输入阶段 Ctrl-C 和异常退出共享清理路径。
- 仅删除仍使用临时标题，且没有 run、消息、动作、任务、来源或 session 级记忆的会话。
- 已发送问题、手动重命名或保存 session 级记忆的会话不会被自动删除。

## 升级与回滚

首次打开 v1.4.0 工作区数据库时，CapsLock 会在 `.capslock/backups/` 创建 schema v2 备份，再升级到 schema v3。迁移为 `sessions` 增加标题、标题来源和标题更新时间，并使用最早的 run 问题回填旧会话标题；没有问题的旧会话使用临时标题。

迁移失败会回滚并保留备份，重复启动不会重复备份。v1.4.0 不支持读取 schema v3；回滚前必须退出 CapsLock，并从 `.capslock/backups/` 恢复 schema v2 数据库。

## 兼容性与边界

- 会话数据库仍位于当前工作区的 `.capslock/capslock.sqlite3`，不会跨工作区汇总历史。
- `capslock sessions` 和恢复选择器默认最多显示 20 条记录。
- 现有完整 Session ID 恢复命令保持有效，并新增唯一 ID 前缀解析。
- 会话标题仅用于展示和选择，不替代内部 Session ID，也不会进入模型上下文。
- `prompt-toolkit` 最低版本从 3.0.0 升至 3.0.52。

## 发布检查

本地 Python 3.11/Linux 已通过 91 项测试，以及 Ruff、依赖一致性、仓库卫生、构建、twine、版本一致性和 wheel 安装冒烟检查。发布使用 `v1.4.1` tag；Linux/macOS、Python 3.11/3.12 的远端质量与安全任务全部通过后，才应创建 GitHub Release。
