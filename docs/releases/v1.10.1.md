# CapsLock v1.10.1 发布说明

v1.10.1 是面向 2.0 的前置解耦重构。它收紧运行时、workflow、模型、生命周期、记忆、配置和数据库迁移器的模块边界，同时保持现有用户行为与数据格式兼容。

## 架构边界

- 新增 `capslock.ports` 中立协议层与 `capslock.bootstrap` 组合根，runtime/tooling 不再依赖应用协调器或具体 SQLite repository。
- 使用共享 `RunInteraction` 维护单次运行的权限模式和审批回调，动作工厂不再依赖 Agent 引用回填。
- workflow 的 work item、run、step 和审批结算改由无 I/O 领域策略计算，repository 仍负责 CAS、事务和 terminal event 原子写入。
- 模型调用使用显式 `ModelRunContext` 与 `ModelRunSession`；流式和非流式路径共享候选规划、数据策略、重试、预算与审计逻辑。
- Agent 运行编排、事件发布、终止清理以及 ToolLoop 的模型步骤和工具调用执行被拆成窄职责组件。

## 生命周期、记忆与基础设施

- `capslock.lifecycle` 转为保持原导入路径兼容的 package，并分离归档校验、脱敏、声明式导入规格与双库失败恢复协调。
- memory repository 拆为 lifecycle、query、source、settings 与 audit 边界；embedding consent、外发预览和缓存清理由独立策略服务管理。
- 配置读取建立 document reader、validator、migrator、settings resolver 管线；数据库升级由 `DatabaseSpec` 与 `MigrationStep` 声明。
- 历史记录中的 `You` 标签使用独立的加粗强调色，避免与 prompt 正文混淆。

## 兼容与升级

本版本不提高配置、workspace 或 memory schema version，不改变 portable archive、事件格式、CLI、退出码或审批权限，也不会修改现有用户数据。`WorkspaceApplication`、`WorkspaceAgent`、`WorkflowService`、`ActionCoordinator`、`ToolLoop`、`ModelRouter`、`MemoryService` 和 `LifecycleService` 的既有导入路径继续有效。

旧 `repositories=`、`max_turns=` 以及 `ModelRouter.bind_run()`、`use_role()`、`summary()` 等 Python 入口仍可调用，并发出默认隐藏、注明最早在 2.0.0 删除的 `DeprecationWarning`。新代码应改用窄端口、`max_tool_rounds` 和显式 `ModelRunSession`。

详细模块边界与完整弃用表见 [v2 架构与迁移说明](../development/v2.md)。
